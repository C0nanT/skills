# `/triage` — Issue Triage

## What it is

A skill that moves issues through a triage role state machine. Lets you manage the flow of bugs and feature requests in a structured way.

## What it's for

- Seeing which issues need attention right now
- Evaluating a specific bug or feature request
- Preparing issues to be executed by an AFK agent
- Managing the issue workflow without manually opening GitHub/GitLab

## How to invoke

In natural language after `/triage`:

```
/triage
Show me anything that needs my attention

/triage
Let's look at #42

/triage
Move #42 to ready-for-agent

/triage
What's ready for agents to pick up?
```

**Prerequisite**: run `/setup-skills` before using for the first time.

## How it works

### State machine

Each triaged issue has exactly one **category** role and one **state** role:

**Categories:**
- `bug` — something is broken
- `enhancement` — new feature or improvement

**States:**
- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting for more information from the reporter
- `ready-for-agent` — fully specified, ready for AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Normal transitions: no label → `needs-triage` → (`needs-info` | `ready-for-agent` | `ready-for-human` | `wontfix`). `needs-info` returns to `needs-triage` when the reporter responds.

### Mandatory disclaimer

Every comment or issue posted during triage starts with:
```
> *This was generated by AI during triage.*
```

### Seeing what needs attention

Presents three buckets, oldest to newest:
1. **No label** — never triaged
2. **`needs-triage`** — evaluation in progress
3. **`needs-info` with recent reporter activity** — needs re-evaluation

### Triaging a specific issue

1. **Collects context** — reads the full body, comments, labels, reporter. Reads prior triage notes to avoid repeating already-answered questions.

2. **Recommends** — states the recommended category and state with justification, plus a summary of the relevant codebase. Waits for your direction.

3. **Reproduces** (bugs only) — before any grilling, tries to reproduce: reads the reporter's steps, traces the relevant code, runs tests. Reports the result.

4. **Grills if needed** — if the issue needs more detail, enters a `/grill-with-docs` session.

5. **Applies the result:**
   - `ready-for-agent` → posts an agent brief (complete instructions for an agent to execute without human context)
   - `ready-for-human` → same format as the agent brief, but notes why it can't be delegated
   - `needs-info` → posts a triage notes template with specific, actionable questions
   - `wontfix` (bug) → polite explanation, closes the issue
   - `wontfix` (enhancement) → writes to `.out-of-scope/`, links from the comment, closes

### Quick state override

If you say "move #42 to ready-for-agent", the skill trusts you and applies directly. Confirms what it will do, then acts. No grilling.

## Usage example

```
/triage
Show me what needs attention
```

The agent lists: "3 issues with no label, 1 in needs-triage, 2 in needs-info with reporter response." You choose which to tackle.

```
/triage
Let's look at #15
```

The agent reads issue #15 (supposed login bug), tries to reproduce, recommends `bug / ready-for-agent` with justification, you confirm, it posts the agent brief.
